## 背景

仓库目前以技术方案文档和招聘相关 Excel 数据为主，还没有 Node.js 应用代码。目标是在竞赛路演阶段快速搭建一个可运行、可演示、可 Docker 化的人力招聘数据平台 MVP，为后续招聘数据导入、候选人管理、渠道分析和人效核算提供稳定基础。

方案文档已明确技术方向：Node.js + Express、EJS + Bootstrap5、SQLite、单容器部署。设计重点是把第一版平台边界收窄到“能启动、能访问、能连接 SQLite 文件、能挂载基础页面”，避免把所有招聘业务逻辑塞进入口文件。

## 目标 / 非目标

**目标：**

- 提供最小可运行的 Express 应用骨架。
- 使用 `server.js` 作为唯一启动入口，完成应用装配和监听。
- 支持 EJS 服务端渲染、Bootstrap5 页面展示、静态资源访问。
- 支持 SQLite 文件模式连接，默认数据库文件路径为 `data/recruitment.db`。
- 提供仪表盘、候选人管理、招聘进度、渠道分析、系统配置的基础路由入口。
- 提供 Docker 单容器运行基础，匹配路演阶段一键启动诉求。

**非目标：**

- 不在本次 MVP 中实现完整账号权限体系。
- 不在本次 MVP 中实现复杂 Excel 导入、清洗和自动分析。
- 不在本次 MVP 中设计候选人、招聘进度、渠道分析等业务表结构。
- 不在本次 MVP 中内置演示数据。
- 不在本次 MVP 中迁移到 PostgreSQL 或引入 Docker Compose。
- 不在 `server.js` 中编写候选人、渠道、人效等具体业务规则。
- 不引入 Vue、React、前端构建工具或前后端分离架构。

## 设计决策

### 决策 1：`server.js` 只做启动装配

`server.js` 负责创建 Express 实例、配置模板引擎、挂载静态资源和中间件、初始化数据库、挂载路由、注册错误处理、监听端口。业务逻辑下沉到 `routes/`、`service/`、`dao/`。

备选方案是把页面处理、SQL 和统计逻辑直接写进 `server.js`。这种方式初期更快，但文件会迅速膨胀，不利于 Cursor 继续生成和维护多页面后台。

### 决策 2：使用 EJS 服务端渲染而不是 SPA

第一阶段使用 EJS + Bootstrap5 生成后台页面。页面数据由后端直接注入模板，避免跨域、接口联调、前端打包和路由配置。

备选方案是 Vue3 或 React SPA。它适合长期产品化，但对当前 5-6 个页面的路演后台而言工程化成本过高。

### 决策 3：SQLite 默认使用文件模式

应用启动时通过数据库配置模块连接 SQLite。默认使用文件模式，数据库文件路径为 `data/recruitment.db`，用于后续需求继续设计页面和表结构；必要时可通过环境变量切换到内存库。

备选方案是直接使用 JSON 文件或内存数组。它们更简单，但无法平滑过渡到正式数据库，也不利于后续 SQL 统计分析。

### 决策 4：路由先提供页面入口，业务逐步填充

MVP 首先提供页面可访问性和导航结构：`/`、`/candidates`、`/progress`、`/channels`、`/settings`。每个路由先返回空状态页面，后续再根据新需求设计真实页面、表结构、service 和 DAO。

备选方案是一次性实现所有页面的完整 CRUD 和统计。该方案范围过大，容易拖慢从 0 到可运行平台的第一步。

### 决策 5：Docker 只承载单应用进程

路演阶段镜像内只运行 Node.js 进程，SQLite 作为嵌入式文件由应用访问。通过数据卷保留数据库文件。

备选方案是在单镜像中同时运行数据库进程，或直接进入 Docker Compose。两者都增加部署和排错成本，不符合当前“一条命令启动”的目标。

## 风险 / 取舍

- 风险：EJS 服务端渲染在交互复杂后可能变得笨重。缓解方式：第一阶段只做后台管理基础交互，复杂图表和导入流程后续可局部增强。
- 风险：SQLite 并发能力弱于正式数据库。缓解方式：路演和轻量内部试用阶段可接受，设计中保留 DAO 层，后续可迁移 PostgreSQL。
- 风险：Bootstrap CDN 在离线演示环境不可用。缓解方式：第一版按确认结果使用 CDN，后续如需离线演示再下载到 `public/vendor/`。
- 风险：入口文件职责失控。缓解方式：将 `server.js` 限定为装配层，新增业务必须放入路由、service 或 DAO。
- 风险：招聘数据中可能包含个人敏感信息。缓解方式：MVP 页面避免展示身份证、手机号等敏感字段；后续正式落地需增加脱敏、权限和加密存储策略。

## 迁移计划

1. 创建 Node.js 项目基础文件和目录。
2. 实现 `server.js`、SQLite 文件连接、基础路由、布局模板和首页。
3. 增加 Dockerfile 和 `.dockerignore`。
4. 本地运行 `npm start` 验证页面可访问。
5. 构建 Docker 镜像并验证容器内启动。

回滚策略：该变更是新增应用骨架，不改动现有数据文件和文档；如需回滚，可删除新增代码目录和 OpenSpec change。
