## Context

当前项目是一个 Node.js 24、Express 5、EJS、SQLite 文件模式的单体 SSR 应用。已有 `server.js`、`routes/index.js`、`dao/db.js`、公共 EJS 布局和 MVP 空状态页面，主规格 `recruitment-platform-mvp` 明确目前只覆盖平台骨架，不包含业务表结构和业务数据。

本次 change 要在该骨架上承载三类源数据：在职/离职员工 Excel、月度招聘目标 Excel、面试记录 Excel，并把现有招聘负责人看板中的目标达成、基地风险、自主社招人效等口径产品化。页面视觉参考用户提供的后台截图：左侧深色可收起导航、两层结构、右侧筛选区和表格区。

约束：

- 首期继续使用现有单体应用和 SQLite，不引入前后端分离架构。
- 文档、页面文案和业务字段使用中文；代码标识、路径和配置保留英文。
- 原始 Excel 来自外部数据源，员工和目标存在覆盖更新场景，面试记录同时存在全量覆盖、按日追加和按日覆盖场景。
- 手机号、证件号等个人敏感信息必须按安全合规要求处理，列表默认展示脱敏值。

## Goals / Non-Goals

**Goals:**

- 将平台从 MVP 空状态扩展为招聘数据分析后台，提供员工数据管理、招聘目标管理、面试记录管理和综合分析看板四个模块。
- 建立统一业务数据模型，支持 Excel 导入、覆盖更新、列表查询、分页和导入结果反馈。
- 复用并沉淀现有招聘目标达成口径，包括基地归类、渠道顺序、月度目标、截止目标、实际入培、GAP、达成率、渠道占比和自主社招人效。
- 为所有列表页提供一致的筛选、查询、重置、导入弹框和分页体验。
- 保留导入记录，便于追踪每次覆盖、追加、失败和校验结果。

**Non-Goals:**

- 不建设登录、权限、审批流或多租户能力。
- 不接入外部 HR 系统、ATS 系统或在线数据库同步。
- 不做复杂图表库选型；首期可用 SSR 表格、统计卡片和轻量静态交互表达分析结果。
- 不实现完整招聘前链路漏斗；在缺少邀约、投递、到面等数据源前，漏斗首期只覆盖面试反馈到入培的后段分析。
- 不支持手工编辑业务数据；首期数据以 Excel 导入为准。

## Decisions

### 1. 沿用单体 SSR 架构

保留 Express + EJS + SQLite 的当前技术路线，在现有路由和页面体系上扩展业务模块。相比引入 React/Vue 管理后台，SSR 更贴合当前代码规模，也能更快交付导入、查询和报表页面。

替代方案是前后端分离，但会引入构建链路、接口契约和状态管理成本；当前需求以数据管理和报表为主，单体 SSR 足够支撑。

### 2. 以标准化表承接原始 Excel

导入时不直接按 Excel 原始列驱动页面，而是进入标准化业务表：

- `employees`：统一员工表，合并在职和离职来源，归一 `部门/职位/员工状态/离职日期` 等字段。
- `recruitment_targets`：目标配置表，保留月份、基地、渠道、订单类型、留存率目标、月度目标和 1 日至 31 日日目标。
- `interview_records`：面试记录表，保留职位、候选人、电话、反馈日期、反馈结果、面试官、渠道标签、合约名称、内推人和综合评价。
- `import_batches`：导入批次表，记录来源类型、导入模式、覆盖范围、文件名、成功行数、失败行数和错误摘要。

该设计让页面查询和指标计算依赖稳定字段，同时保留外部文件格式变化的校验边界。

### 3. 覆盖更新使用事务和批次边界

员工导入以来源类型为边界全量覆盖：在职员工表覆盖 `source_type = active` 的员工记录，离职员工表覆盖 `source_type = resigned` 的员工记录。目标导入以 `year_month` 为边界覆盖同月目标。面试记录支持三种模式：全量覆盖、按日追加、按日覆盖。

所有覆盖操作必须在 SQLite 事务内完成，导入失败时回滚并保留失败批次信息，避免出现半导入数据。

### 4. 角色列表基于同一员工查询模型

招聘专员列表和一线员工列表复用员工列表查询、分页和表格列，只通过角色口径筛选：

- 招聘专员：`position = 招聘专员`。
- 一线员工：职位属于市场一线岗位口径；首期以 `客服专员`、`培训期学员` 等员工数据中稳定出现的岗位为默认候选，并在代码中集中维护口径，便于后续调整。

这样可以避免两个页面复制字段、分页和筛选逻辑。

### 5. 指标计算沉淀为服务层

招聘目标达成、基地归类、渠道归一、人效核算和漏斗分析不写在 EJS 模板里，而是放入服务层。页面只接收已计算好的 DTO：

```text
Excel 导入 -> 标准表 -> 查询/指标服务 -> EJS 页面
```

服务层优先复用现有 `recruitment-progress-report` 口径：实际入培合并在职和离职后按工号去重，按入培时间筛选；渠道顺序固定为 `回流`、`内推`、`渠道社招`、`渠道校招`、`自主社招`、`合计`；百分比字段分母为 0 时展示 `0.00%`。

### 6. 统一后台布局和列表组件

公共布局从顶部 Bootstrap 导航调整为左侧两层导航。列表页统一使用：

- 筛选区：关键词、基地、职位、渠道、状态、日期范围等按页面配置展示。
- 操作区：查询、重置、导入。
- 导入弹框：文件选择、导入模式、覆盖范围提示、校验结果和导入结果。
- 表格区：默认每页 20 条，支持分页。

首期可以通过 EJS partial 组织筛选区、分页和导入弹框，避免引入前端组件框架。

## Risks / Trade-offs

- [Risk] 原始 Excel 字段发生变化导致导入失败 → Mitigation：每类导入维护必填列校验，导入前返回缺失列和异常行摘要。
- [Risk] 一线员工“任意市场岗位”口径不够精确 → Mitigation：将一线岗位口径集中配置，首期按现有数据默认值实现，并在页面说明当前口径。
- [Risk] 手机号和证件号涉及个人敏感信息 → Mitigation：数据库存储前预留加密/脱敏处理边界，列表默认展示脱敏手机号，不在日志输出完整敏感字段。
- [Risk] 面试记录无法支撑完整招聘漏斗 → Mitigation：首期明确为“面试反馈到入培”的后段漏斗，等待新增数据源后再扩展前链路。
- [Risk] 大批量 Excel 导入阻塞请求 → Mitigation：首期数据量可由同步导入承载；导入过程使用事务，页面显示处理中状态。若后续文件明显增大，再拆分为异步任务。
- [Risk] 目标和员工数据的基地/渠道口径不一致 → Mitigation：服务层集中做基地归类、渠道归一和未匹配数据输出，分析页展示未匹配提醒。

## Migration Plan

1. 在现有 SQLite 初始化中新增业务表和索引，保留已有数据库文件路径配置。
2. 新增导入服务，先支持 Excel 解析、字段校验、事务覆盖和批次记录。
3. 将 MVP 空状态路由替换为四个业务模块页面，并保留旧路由到新页面的兼容跳转或空状态说明。
4. 接入员工、目标、面试列表页面，再接入目标达成和综合分析页面。
5. 使用用户提供的 2026 年员工、目标、面试样例文件进行端到端验证。

回滚策略：由于首期不修改外部系统，回滚可通过恢复应用代码并保留或删除新增 SQLite 业务表完成。导入操作以批次记录为边界，可通过重新导入最近一次可信 Excel 文件恢复业务数据。

## Open Questions

- 一线员工岗位口径是否需要只包含 `客服专员` 和 `培训期学员`，还是包含班长、主管、质检、培训等运营岗位？
- 手机号是否允许管理员在详情页查看完整值，还是全后台均只展示脱敏值？
- 目标导入是否需要支持同一个月多个历史版本查看，还是只保留当前生效版本和导入记录？
- 面试记录按日追加时，重复数据的唯一键是否采用 `电话 + 职位名称 + 面试官填写反馈时间`？
