# 人力招聘数据分析后台技术方案文档

## 一、需求总结

### 1\.1 项目背景

本项目为公司内部 AI 开发竞赛参赛作品，面向人力资源岗位打造专属内部管理后台，核心用于招聘全流程数据的统一管理与可视化分析。项目由非技术岗位人员基于 Cursor AI 工具辅助开发，以轻量化、易落地、可演示为核心建设原则。

### 1\.2 核心目标

1. 实现招聘数据的集中管理，包含候选人信息、招聘进度、渠道来源等核心数据的录入、查询与维护

2. 提供基础数据分析能力，直观呈现招聘漏斗、渠道效率、人员分布等关键业务指标

3. 以 Docker 镜像形式交付，一键部署即可使用，满足竞赛路演的演示需求

4. 架构预留扩展空间，支持后续从演示版本平滑升级为正式内部持久化服务

### 1\.3 功能范围

项目规划 5\-6 个核心页面，覆盖内部后台完整基础能力：

- 数据概览仪表盘：招聘核心指标、趋势图表展示

- 候选人管理：候选人信息列表、新增、编辑、详情查看

- 招聘进度管理：面试流程、录用状态跟踪

- 渠道数据分析：各招聘渠道的转化效果统计

- 系统基础配置：字典项、账号权限等基础设置

### 1\.4 部署与阶段规划

- **第一阶段（竞赛路演）**：单容器一体化部署，开箱即用，快速启动演示，数据支持临时存储与可选本地持久化

- **第二阶段（正式落地）**：平滑升级为独立数据库 \+ 应用服务的标准架构，支持多用户访问、数据持久备份、正式运维

### 1\.5 约束条件

1. 开发人员非专职技术岗，依赖 AI 工具辅助编码，技术栈需低门槛、高通用性、AI 生成准确率高

2. 最终交付为单个 Docker 镜像，部署操作极简，无需复杂环境配置

3. 路演阶段服务器资源有限，镜像体积与运行资源占用需轻量化

4. 代码结构清晰，便于后续迭代维护与功能扩展

## 二、技术选型考量

### 2\.1 选型总原则

所有技术选型均围绕四大核心原则：**低开发门槛、AI 生成友好、轻量化易部署、可平滑演进**，优先选择生态成熟、文档完善、语法简单的技术方案，最大化降低非技术人员的开发难度。

### 2\.2 后端技术选型

**选型结论：Node\.js \+ Express 框架**

#### 选型理由

1. **AI 开发友好度最高**：Node\.js 是 Cursor 等 AI 代码工具支持最完善的后端语言，全栈代码生成连贯性强，从接口到页面模板可一次性生成，上下文断层少，非技术人员调试成本最低。

2. **前后端语言统一**：前端交互脚本与后端服务均使用 JavaScript，无需切换多门语言，降低学习与理解成本。

3. **生态成熟完善**：Express 是最经典的轻量 Web 框架，招聘后台所需的路由、表单处理、静态资源、模板渲染等能力全部开箱即用，社区示例极其丰富。

4. **镜像轻量化**：基于 alpine 的 Node\.js 基础镜像仅 110MB，无重型运行时依赖，构建与拉取速度快。

#### 备选方案对比

- Python \+ Flask：语法简单但前后端语言不统一，AI 生成全栈代码时易出现前后端衔接问题，且数据分析相关依赖会显著增大镜像体积

- Java SpringBoot：架构过重，配置繁琐，镜像体积大，非技术人员学习成本极高，完全不匹配本项目场景

### 2\.3 数据库技术选型

**选型结论：路演阶段采用 SQLite（内存 / 文件双模式兼容），正式阶段可平滑迁移至 PostgreSQL**

#### 选型理由

1. **零配置开箱即用**：无需独立安装数据库服务、无需配置账号密码、无需启动额外进程，一行代码即可初始化数据库，非技术人员零运维成本。

2. **单容器完美适配**：数据库作为文件库嵌入应用，无需多容器编排，单个镜像即可完整运行，完美匹配路演一键部署需求。

3. **AI 生成友好**：SQL 语法标准通用，Cursor 可直接生成对应增删改查代码，学习成本远低于其他数据库。

4. **平滑演进能力**：采用标准 SQL 语法开发，后续切换至 PostgreSQL 仅需修改连接配置，核心业务代码无需大规模重构。

5. **双模式灵活切换**：内存模式启动速度最快，适合路演演示；文件模式可配合 Docker 数据卷实现数据持久化，适合临时正式使用。

#### 备选方案对比

- PostgreSQL：功能强大但需独立服务进程，单容器部署需额外配置启动脚本，镜像体积增加 260MB 以上，配置项多、出错概率高，不适合路演快速交付

- MySQL：同样需独立服务，镜像体积更大，配置更繁琐，轻量场景下性价比远低于 SQLite

### 2\.4 前端技术选型

**选型结论：EJS 服务端模板引擎 \+ Bootstrap5 UI 组件库**

#### 选型理由

1. **零工程化配置**：无需前端构建工具、无需打包编译、无需配置路由与环境，写完页面直接生效，非技术人员可快速上手，完全避免 Vue/React 等框架的工程化配置门槛。

2. **开发效率极高**：5\-6 个简单页面的场景下，服务端渲染模式开发速度远高于前后端分离架构，AI 可直接生成完整页面代码，无需处理跨域、接口联调等问题。

3. **UI 组件开箱即用**：Bootstrap5 内置完整的后台管理组件，表格、表单、弹窗、导航、卡片、图表容器一应俱全，无需编写自定义 CSS 即可产出规范的后台界面。

4. **镜像零额外体积**：所有前端代码均为纯文本模板与 CDN 引入资源，不增加镜像体积，维持整体轻量化特性。

5. **与后端无缝衔接**：EJS 与 Express 原生兼容，数据直接从后端注入页面，无需单独开发 API 接口，开发流程更简洁。

#### 备选方案对比

- Vue3/React 单页应用：需要工程化构建、依赖管理、路由配置，额外增加大量学习成本与镜像体积，5\-6 个页面的场景下 90% 的能力闲置，属于过度选型

- 原生 HTML\+CSS：样式开发效率低，界面规范度差，重复代码多，维护成本高

### 2\.5 容器化部署选型

**选型结论：单容器一体化镜像（路演阶段），预留 Docker Compose 扩展能力（正式阶段）**

#### 选型理由

1. **交付与演示成本最低**：单个镜像文件即可完整运行，部署仅需一条 docker run 命令，无需编排配置，路演场景下迁移、演示、分发都最便捷。

2. **运行资源占用低**：单进程运行，内存占用小，适配公司服务器有限的资源环境。

3. **可平滑升级**：后续正式部署时，可将数据库拆分出独立容器，通过 Docker Compose 编排，应用代码无需改造。

## 三、整体架构设计

### 3\.1 逻辑分层架构

项目采用经典的五层分层架构，职责清晰，便于 AI 生成代码与后续维护，各层单向依赖，结构规整。

|层级|职责说明|技术实现|
|---|---|---|
|表现层|页面渲染、用户交互、数据展示|EJS 模板 \+ Bootstrap5 \+ 原生 JavaScript|
|接口层|路由分发、请求参数校验、响应返回|Express Router|
|业务逻辑层|招聘数据统计、流程规则处理、业务计算|业务 Service 模块|
|数据访问层|数据库增删改查、SQL 封装、数据持久化|SQLite 数据库操作模块|
|数据层|数据存储|SQLite 内存库 / 本地文件库|

### 3\.2 单容器部署架构

路演阶段采用单容器一体化部署架构，所有组件运行在同一个 Docker 容器内，对外仅暴露一个访问端口。

```Plain Text
┌─────────────────────────────────────┐
│           Docker 容器               │
│  ┌───────────────────────────────┐  │
│  │    Node.js 运行环境           │  │
│  │  ┌─────────┐  ┌───────────┐  │  │
│  │  │ Express │  │ 业务代码   │  │  │
│  │  │ 服务    │  │ 与模板     │  │  │
│  │  └────┬────┘  └─────┬─────┘  │  │
│  │       └──────┬──────┘        │  │
│  │              │               │  │
│  │        SQLite 数据库         │  │
│  │  （内存模式 / 文件模式）     │  │
│  └───────────────────────────────┘  │
│                 │                    │
└─────────────────┼────────────────────┘
                  │ 3000端口
          ┌───────┴───────┐
          │ 浏览器访问     │
          └───────────────┘
```

#### 架构说明

1. 容器内仅运行 Node\.js 一个主进程，SQLite 作为嵌入式库直接调用，无额外数据库进程，稳定性高

2. 所有页面与接口均通过 3000 端口访问，无跨域问题，无需额外配置

3. 文件模式下，可通过 Docker 数据卷挂载数据库文件，实现容器销毁数据不丢失

4. 镜像整体体积约 120MB，启动时间小于 1 秒，路演演示体验流畅

### 3\.4 核心运行流程

容器启动后自动完成全流程初始化，无需人工干预，实现真正的一键启动即用。

1. **容器启动**：执行 Node 入口文件，初始化 Express 服务

2. **数据库初始化**：自动连接 SQLite 数据库，执行建表语句，可选注入演示数据

3. **服务启动**：Express 服务监听 3000 端口，加载所有路由与静态资源

4. **访问使用**：用户通过浏览器访问 3000 端口，后端渲染页面并返回

5. **数据交互**：用户操作页面提交数据，后端完成业务处理与数据库读写，返回结果

### 3\.5 后续持久化演进方案

项目从路演版升级为正式持久化版本时，采用平滑演进方案，核心业务代码复用率可达 90% 以上。

1. **数据库升级**：将 SQLite 文件库数据迁移至 PostgreSQL 独立容器，仅修改数据库连接配置与少量方言 SQL，业务逻辑代码无需改动

2. **架构拆分**：采用 Docker Compose 编排，拆分为 Node\.js 应用容器、PostgreSQL 数据库两个独立容器，实现资源隔离与独立扩缩容

3. **功能扩展**：可新增用户权限体系、数据备份策略、日志监控等正式环境必备能力

4. **部署升级**：支持反向代理、HTTPS 配置、多实例部署等生产级特性，适配公司正式服务器环境
