## §1 平台架构一句话

```
你的电脑（写代码 + docker build + docker push）
        │
        ▼
   平台 ACR（每队一个私有仓库，跟队伍编号绑定）
        │
        ▼
   平台 K8s（收到你的部署请求 → 拉镜像 → 挂载数据库 Secret → 启动 Pod → 挂 Ingress + 域名）
        │
        ▼
   评委 / AI 评估 通过 hack-team-XXX.jiarui.net.cn 访问你的应用
```

平台**只做**这些事：

- 给你一个专属 PostgreSQL 数据库 + 自动挂载凭证到容器
- 给你一个专属 ACR 仓库 + 一个 K8s namespace
- 部署时按你的镜像 tag 起 pod、配 Ingress、发独立域名

平台**不做**这些事：

- ❌ 不构建你的镜像（`docker build` 在你自己的电脑上跑）
- ❌ 不注入除数据库以外的任何环境变量（API key、feature flag、三方 token 等，都靠 Dockerfile 或代码自带）
- ❌ 不挂载文件系统 / 卷（除了 emptyDir，容器重启数据就没了 —— 持久化的东西都塞进数据库）
- ❌ 不做定时任务 / 后台 worker（一次部署 = 一个 Deployment，就是一个 Web 进程）

## §2 数据库配置（本地 vs 线上）

### §2.1 每队会收到什么

每个队伍会拿到**一套独立的 PostgreSQL 数据库**（阿里云 RDS，Zhangjiakou 区）：

| 属性                                 | 值                                                         |
| ------------------------------------ | ---------------------------------------------------------- |
| host（**公网**，本地开发用）   | `pgm-8vb9x4vt2nmk1kcc1o.pgsql.zhangbei.rds.aliyuncs.com` |
| host（**VPC 内网**，K8s 里用） | `pgm-8vb9x4vt2nmk1kcc.pgsql.zhangbei.rds.aliyuncs.com`   |
| port                                 | `5432`                                                   |
| dbname / user / password             | 每队独有，会通过平台后台看到（或管理员在群里发）           |

**两个 host 的区别只是网络路径 —— 数据库是同一个**。本地开发走公网稍慢；容器里走内网又快又不花外网流量。

### §2.2 本地开发怎么配

**唯一的原则**：应用代码只读 `process.env.DATABASE_URL`（Python 用 `os.environ["DATABASE_URL"]`）。**别在代码里硬编码任何东西**。

在项目根目录建 `.env`：

```
DATABASE_URL=postgresql://team-c06-user:PASSWORD@pgm-8vb9x4vt2nmk1kcc1o.pgsql.zhangbei.rds.aliyuncs.com:5432/team_c06
```

（真实值用平台发给你的队伍那套凭证）

**必须** 在同级目录建 / 更新两个文件：

```


# .gitignore
.env

# .dockerignore
.env
```

这两个文件确保：

- `.env` 不进 git（凭证不泄露）
- `.env` 不进 Docker 镜像（Dockerfile 里 `COPY . .` 的时候 `.env` 会被跳过）

**为什么 `.dockerignore` 里也必须有 `.env`**：如果 `.env` 被 `COPY` 进镜像，容器启动时会用镜像里的公网 DSN，把平台注入的内网 DSN 覆盖掉 —— 应用要么慢 3 倍，要么直接连不上（VPC 出网走公网 host 有时会因防火墙被拒）。**这个坑是本次活动最高频的 bug**。

框架层怎么读：

```
// Next.js / Node.js —— 用 dotenv 或 next 的内建加载
const dsn = process.env.DATABASE_URL;

// Prisma —— schema.prisma 里
// datasource db { url = env("DATABASE_URL") }

// 用 pg
import { Pool } from "pg";
const pool = new Pool({ connectionString: process.env.DATABASE_URL });
```

```


# Python
import os
DATABASE_URL = os.environ["DATABASE_URL"]

# SQLAlchemy
from sqlalchemy import create_engine
engine = create_engine(os.environ["DATABASE_URL"])

# psycopg
import psycopg
conn = psycopg.connect(os.environ["DATABASE_URL"])
```

### §2.3 部署到平台后发生了什么

你 `docker push` 完，去「部署」页面点部署。平台在你的 K8s namespace 里做了两件事：

1. **确保 `team-db` Secret 存在**（包含 `DATABASE_URL / DB_HOST / DB_PORT / DB_NAME / DB_USER / DB_PASSWORD` 六个 key，全部指向 **VPC 内网** host）
2. **给你的 Deployment 加 `envFrom`**，把这个 Secret 里的所有 key 作为环境变量注入到容器 —— 你的应用**开箱就能读到 `DATABASE_URL`**，且是内网版

关键：**你的应用感知不到自己是"本地"还是"线上"**。同一份代码、同一份 Dockerfile，两处都跑得起来。**这个透明切换的前提就是"代码里只读 env，不硬编码"**。

### §2.4 6 个可用变量清单（部署后自动就有）

| 变量             | 内容                                                          | 常用度              |
| ---------------- | ------------------------------------------------------------- | ------------------- |
| `DATABASE_URL` | `postgresql://user:pass@INTERNAL_HOST:5432/dbname` 完整 DSN | ⭐⭐⭐ 用这个就够了 |
| `DB_HOST`      | 只有 host（内网）                                             | 需要自己拼 DSN 时用 |
| `DB_PORT`      | 5432                                                          | 同上                |
| `DB_NAME`      | 数据库名                                                      | 同上                |
| `DB_USER`      | 用户名                                                        | 同上                |
| `DB_PASSWORD`  | 密码                                                          | 同上                |

99% 的场景直接读 `DATABASE_URL` 就好，剩下 6 个是给某些框架非要拆开的场景用（如 Django settings.py 里的 `DATABASES = { "default": { "NAME": ... } }`）。

### §2.5 本地怎么模拟"线上环境"

想在部署前先在本机模拟一遍容器里的行为，用 `-e` 传：

```
docker run --rm \
  -e DATABASE_URL="postgresql://user:pass@公网host:5432/dbname" \
  -e OPENAI_API_KEY=sk-xxx \
  -p 3000:3000 \
  你的镜像tag
```

看到应用正常起来、能查库，说明"平台注入 env → 应用启动"这套链路是通的。**任何本地跑不起来的问题，都别急着往平台上传** —— 先在本机 debug 干净。

## §3 环境变量的取舍（**除了 DB，全部塞进 Dockerfile**）

### §3.1 硬规则

- ✅ **DB 6 个变量** → 平台注入，Dockerfile 里**不用写**
- ❌ **其他所有变量** → 平台**不注入**，你必须在 Dockerfile 里 `ENV xxx=yyy` 硬编码

这个二分法是平台**为了 48 小时活动做简单化**的设计 —— K8s 只挂了一个 Secret（`team-db`），没有给参赛者预留额外的 Secret 挂载或 Deployment env 注入通道。想改这个规则得改平台代码，来不及。

### §3.2 哪些是"其他变量"

| 类别            | 举例                                                                                                                                                    |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| LLM API key     | `OPENAI_API_KEY` / `ANTHROPIC_API_KEY` / `DEEPSEEK_API_KEY` / `GLM_API_KEY` / `GOOGLE_API_KEY`                                                |
| 三方服务        | `SUPABASE_URL` / `SUPABASE_ANON_KEY` / `SENTRY_DSN` / `REDIS_URL` / `OSS_ACCESS_KEY`                                                          |
| 应用配置        | `NODE_ENV=production` / `NEXT_TELEMETRY_DISABLED=1` / `LOG_LEVEL=info` / `TZ=Asia/Shanghai`                                                     |
| 前端 build-time | `NEXT_PUBLIC_*` / `VITE_*` / `REACT_APP_*`（这些其实是在 `npm run build` 时被读进 JS bundle，`ENV` 得在 `npm run build` **之前** 写） |

Dockerfile 完整例子：

```
FROM node:20-alpine
WORKDIR /app

# 国内镜像源（详见镜像加速速通文档）
RUN npm config set registry https://registry.npmmirror.com

# ⭐ 你的应用需要的所有非 DB 环境变量
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1
ENV TZ=Asia/Shanghai
ENV OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxxxx
ENV DEEPSEEK_API_KEY=sk-yyyyyyyyyyyyyyyyyy


# 如果你用了三方服务
ENV SUPABASE_URL=https://xxxxx.supabase.co
ENV SUPABASE_ANON_KEY=eyJhbGciOi...


# 前端 build-time 变量必须写在 build 前
ENV NEXT_PUBLIC_APP_URL=https://hack-team-XXX.jiarui.net.cn

# 装依赖 + build
COPY package.json package-lock.json* ./
RUN npm ci
COPY . .
RUN npm run build

EXPOSE 3000
CMD ["npm", "start", "--", "-H", "0.0.0.0", "-p", "3000"]

# ⚠️ 数据库凭证（DATABASE_URL 等）不用写！平台会自动注入
```

### §3.3 安全提示

镜像会推到平台的**私有 ACR**（只有你自己队伍的凭证能拉、平台的 imagePullSecret 能拉，任何其他人都看不到）。所以 API key 写在 Dockerfile 里在这个活动里**是可接受的**。

但作为好习惯：

- ⚠️ **不要把带 key 的镜像 push 到 Docker Hub / GitHub 公开镜像仓库**
- ⚠️ **不要把带 key 的 Dockerfile 提交到公开 git 仓库**（私有仓库 OK）
- ⚠️ **活动结束后建议轮换一次 API key**

### §3.4 前端 build-time 变量的常见陷阱

Next.js / Vite / CRA 里，`NEXT_PUBLIC_*` / `VITE_*` / `REACT_APP_*` 在 `npm run build` 时会**被硬编码进 JS bundle**（因为浏览器端读不到 `process.env`）。这意味着：

- ✅ 必须在 Dockerfile 的 **`RUN npm run build` 之前** `ENV NEXT_PUBLIC_XXX=yyy`
- ❌ 只在运行时（`CMD` 之后）注入，浏览器端拿到的是 `undefined`

所以：**前端配置全部走 Dockerfile 里的 `ENV`，跟后端 key 一样对待**。

## §4 Docker 迭代节奏

### §4.1 单次迭代流程

```


# 1. 改代码
vim src/xxx

# 2. build 一个新 tag
docker build -t my-app:v3 .

# 3. 本地跑一遍验证（务必！）
docker run --rm -e DATABASE_URL="公网 DSN" -p 3000:3000 my-app:v3

# 4. 打成平台 ACR 需要的 tag
docker tag my-app:v3 jzharbor-registry.cn-zhangjiakou.cr.aliyuncs.com/hack/team-XXX:v3

# 5. push
docker push jzharbor-registry.cn-zhangjiakou.cr.aliyuncs.com/hack/team-XXX:v3

# 6. 去平台「部署」页面填 v3 点部署
```

### §4.2 为什么每次都用新 tag

- ✅ **回滚快**：v3 挂了 → 回填 v2 立即回滚
- ✅ **cache 生效**：不同 tag = 不同 image ID，K8s 一定会拉新版；同 tag 不同内容会遇到 imagePullPolicy 缓存问题
- ✅ **调试历史清楚**：部署历史里能看到每一版部署时间和状态

**约定成俗**：`v1 v2 v3 …`（简单版本号）或 `20260704-1010`（时间戳）都行，别用 `latest`。

### §4.3 加速二次 build

- `.dockerignore` 里加：`node_modules`、`.git`、`.next`、`dist`、`__pycache__`、`*.log`、`.env`
- 依赖装完再 `COPY . .`（先 `COPY package.json` + `RUN npm ci`，再 `COPY .`）—— 这样只改业务代码时 Docker 会复用依赖层
- 大文件（视频、图片）别塞进镜像 —— 放 OSS 或数据库 bytea

### §4.4 卡在 `docker pull` 的处理

见《[镜像加速速通](https://hack-platform.jiarui.net.cn/docs/mirror)》。三分钟配好 Docker 加速。**48 小时里，头 5 分钟做完这件事，能省你 10 小时。**

## §5 一个镜像跑多个进程 —— 想上 docker-compose 时先看这里

> **谁要看**：想拿 docker-compose 把前端、后端、Redis、后台 worker 拆多个容器的团队 **一句话**：平台**不支持 docker-compose**，一个部署 = 一个镜像。但**一个镜像里跑多个进程**完全可以，`supervisord` 或 `s6-overlay` 3 分钟搞定，AI 工具一句话就能生成模板。

### §5.1 为什么不支持 compose

平台的 K8s 抽象是"一个队 = 一个 Deployment + 一个 Service + 一个 Ingress"，48 小时活动为了简单和稳定性没有做 compose → K8s 的翻译层。想临时加也来不及（活动前一天），基础设施出问题很难 hotfix。**下届活动再考虑**。

### §5.2 你以为需要 compose 的场景，其实单镜像都能搞

| 场景               | 单镜像做法                                                                        |
| ------------------ | --------------------------------------------------------------------------------- |
| 前端 + 后端        | Next.js / Nuxt / Django / Rails 天然一体；或者 nginx 反代 + Node 都塞进一个镜像   |
| App + Redis        | 同镜像里`apt install redis-server`，`supervisord` 起两个进程                  |
| App + 后台 worker  | Node cluster / concurrently，Python multiprocessing，或`supervisord` 起两组进程 |
| 需要"第二个数据库" | 平台已给 PostgreSQL；非要 Redis / SQLite / MongoDB，同容器里跑本地实例            |
| 定时任务           | 应用启动时挂 setInterval / APScheduler，或用`supervisord` 里的 cron 进程        |

### §5.3 例子：Node.js 应用 + Redis 单镜像

Dockerfile：

```
FROM node:20-slim
WORKDIR /app

# 国内镜像加速 + 装 redis 和 supervisord
RUN sed -i 's|http://deb.debian.org|http://mirrors.aliyun.com|g' /etc/apt/sources.list.d/debian.sources \
 && apt-get update && apt-get install -y --no-install-recommends redis-server supervisor \
 && rm -rf /var/lib/apt/lists/*
RUN npm config set registry https://registry.npmmirror.com

ENV NODE_ENV=production
ENV TZ=Asia/Shanghai
ENV OPENAI_API_KEY=sk-替换成你的


# 应用连本地 redis，不跨主机
ENV REDIS_URL=redis://localhost:6379


# 数据库凭证 (DATABASE_URL 等) 平台自动注入，不用写

COPY package.json package-lock.json* ./
RUN npm ci
COPY . .
RUN npm run build

COPY supervisord.conf /etc/supervisor/conf.d/app.conf

EXPOSE 3000
CMD ["/usr/bin/supervisord", "-c", "/etc/supervisor/conf.d/app.conf", "-n"]
```

`supervisord.conf`：

```
[supervisord]
nodaemon=true
user=root
logfile=/dev/null
logfile_maxbytes=0

[program:redis]
command=/usr/bin/redis-server --appendonly no --save "" --bind 127.0.0.1
autorestart=true
priority=10
stdout_logfile=/dev/stdout
stdout_logfile_maxbytes=0
stderr_logfile=/dev/stderr
stderr_logfile_maxbytes=0

[program:app]
command=node /app/build/server.js
directory=/app
autorestart=true
priority=20
stdout_logfile=/dev/stdout
stdout_logfile_maxbytes=0
stderr_logfile=/dev/stderr
stderr_logfile_maxbytes=0
```

要点：

- **`nodaemon=true`**：supervisord 在前台跑，符合容器"一个前台进程"的约定
- **`bind 127.0.0.1`**：Redis 只监听本地，容器外面看不到，也不用担心暴露
- **`priority`**：数字小的先起（redis=10 先起来，node=20 再连）
- **日志全部走 stdout/stderr**：`docker logs` / K8s pod logs 能一次看到两个进程的输出

### §5.4 让 AI 帮你写

把上面这段整个丢给 Claude Code / Codex / ZCode，说"我需要在一个镜像里同时跑 X 和 Y，参照这个模板给我改成 X + Y"，30 秒出结果。**这类模板化的重构是 AI 最擅长的场景。**

替代方案：`s6-overlay`（[github.com/just-containers/s6-overlay](https://github.com/just-containers/s6-overlay)）比 supervisord 更轻，也更容器友好；Python 项目里 `honcho` 用法跟 supervisord 类似。喜欢哪个用哪个。

### §5.5 数据不会持久化 ⚠️

容器重启（每次 `docker push` + 部署都是一次重启）意味着 **Redis / 本地 SQLite / `/tmp` 里的一切数据都会没**。持久化的东西全部放平台给你的 PostgreSQL。48 小时活动够短，Redis 当纯缓存 / session store 用没问题，**但不要把用户数据只写 Redis**。

## §6 AI 对话记录怎么拿到

三种平台"能自动解析"的工具：**Claude Code / Codex / ZCode**。你上传 `.jsonl` 文件，平台会自动转成 markdown，评委和 AI 评委都能看。

### §6.1 Claude Code

对话记录位于：

```
Mac / Linux :  ~/.claude/projects/<你的项目路径>/*.jsonl
Windows     :  %USERPROFILE%\.claude\projects\<你的项目路径>\*.jsonl
```

举例：如果项目在 `/Users/wang/dev/my-app`，那对话文件在：

```
~/.claude/projects/-Users-wang-dev-my-app/
```

（Claude 会把 `/` 换成 `-`，前面加个 `-`）

命令行验证：

```
ls -laht ~/.claude/projects/ | head
ls -laht ~/.claude/projects/-Users-wang-dev-my-app/*.jsonl | head
```

**每个 session 是一个 `.jsonl` 文件，文件名是 UUID**。48 小时里跑几个 session 就传几个文件。

### §6.2 Codex CLI

```
Mac / Linux :  ~/.codex/sessions/年/月/日/rollout-*.jsonl
Windows     :  %USERPROFILE%\.codex\sessions\年\月\日\rollout-*.jsonl
```

举例：

```
ls -laht ~/.codex/sessions/2026/07/04/rollout-*.jsonl
```

⚠️ **不要给 Codex 加 `--no-persist` 参数**！那会跳过日志保存，评委就看不到你的过程，"AI 原生程度"这个维度直接没分。

### §6.3 ZCode

```
Mac / Linux :  ~/.zcode/cli/rollout/model-io-sess_*.jsonl
Windows     :  %USERPROFILE%\.zcode\cli\rollout\model-io-sess_*.jsonl
```

### §6.4 上传前的自检

在自己电脑上：

```


# 快速看一下文件不空、大小合理（一场几小时的对话通常几 MB 到几十 MB）
ls -lah <路径>/*.jsonl

# 看第一条消息是不是你想传的
head -c 500 <某个 .jsonl 文件>

# 大概有几轮对话
wc -l <某个 .jsonl 文件>
```

上传到平台后：

- 平台会自动跑 `convert_transcript` 把 `.jsonl` 转成 markdown → OSS
- 状态从 `pending` → `success` 就代表转换完成
- 评委的判分页面能看到 markdown 化后的整段对话（我们过滤掉了内部"思考"和 skill 调用等对评委没意义的段落）

### §6.5 其他工具

如果你用了 Cursor / Windsurf / Aider / OpenCode 之类，**平台没有针对性解析器**。你需要：

1. 自己把对话导出成 markdown（大部分工具都有导出功能）
2. 在平台上按 `.jsonl` 位置传 `.md`（平台会兜底当纯文本处理）

**但强烈建议**用 Claude Code / Codex / ZCode 之一，因为解析器能把每条消息、tool 调用、错误信息都结构化拿出来，评审看到的信息量更完整。

## §7 一份"能上线跑的项目"的目录长这样

```
my-app/
├── .env                          # 本地公网 DSN + API key（不进 git，不进镜像）
├── .env.example                  # 供队友抄的模板（进 git）
├── .gitignore                    # 至少要有 .env / node_modules / .next
├── .dockerignore                 # 至少要有 .env / node_modules / .next / .git
├── Dockerfile                    # 应用 build + ENV OTHER_KEYS + CMD
├── package.json
├── prisma/
│   └── schema.prisma             # datasource db { url = env("DATABASE_URL") }
├── src/
│   ├── db.ts                     # export const db = new PrismaClient()
│   └── ...
└── README.md
```

**测试验证说明**（`test_instructions`）不放在项目里 —— 直接在平台的「提交」页面填。

## §8 我最想强调的三件事

1. **`.env` 不要 `COPY` 进镜像**。三个位置都要加 `.env`：`.gitignore` / `.dockerignore` / Dockerfile 里也别写 `COPY .env`。这一步 90% 的 DB 连接问题都能省掉。
2. **除了 DB 凭证，其他环境变量都要靠 Dockerfile 里 `ENV` 硬编码**。平台不给注入通道，别指望活动开始后再想办法接入外部 secret 管理。写 Dockerfile 时**把所有 key 都列上**，一次到位。
3. **对话记录别删别改**。找到 `.jsonl` 文件的路径，**7-04 之前**跑一次"打开工具聊两句 → 找到那个 jsonl → 记下路径"的演练，比赛当天直接照着传。

## 附录：一份可以直接抄的最小 Dockerfile

```
FROM node:20-alpine
WORKDIR /app

# 加速
RUN npm config set registry https://registry.npmmirror.com

# 环境变量（真实值替换）
ENV NODE_ENV=production
ENV TZ=Asia/Shanghai
ENV OPENAI_API_KEY=sk-替换成你的

# 数据库凭证不用写！平台会自动注入 DATABASE_URL 等 6 个变量

# 装依赖
COPY package.json package-lock.json* ./
RUN npm ci

# 拷代码 + build
COPY . .
RUN npm run build

# 监听 0.0.0.0，端口 3000
EXPOSE 3000
CMD ["npm", "start", "--", "-H", "0.0.0.0", "-p", "3000"]
```
