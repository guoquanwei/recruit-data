# 伽睿 AI 造物节

## 镜像构建与部署指南

> **谁要看**：技术同事 / 独立参赛者 **什么时候看**：写完应用准备首次部署的时候，或者部署失败排查的时候 **不看的后果**：build 出来的镜像跑不起来 / push 卡住 / 部署后 readiness 检查永远失败

---

## §1 一句话流程

```
本地写代码 → docker build → docker push 到平台 ACR → 在平台「部署」页填镜像 tag → 几十秒内访问 hack-team-XXX.jiarui.net.cn
```

平台不会构建你的镜像，**你自己 build + push**。平台只负责按你提供的 tag 在 K8s 里跑这个镜像。

---

## §2 应用必须满足的两件事

平台用同一套 K8s manifest 跑所有队伍的镜像，因此你的应用必须：

1.  **监听 0.0.0.0**（不是 `127.0.0.1` / `localhost`）—— 否则容器外面看不到端口
2.  **`GET /` 返回 200**（或某个 200 状态的页面）—— 平台的 readiness probe 会一直探这个路径；如果 30 次都不 200，部署就一直 NotReady

> 默认端口是 80。如果你的应用监听别的端口（3000 / 8000 / 8080 / …），在部署表单里填上对应端口号，平台会用那个端口探活。

## §3 Dockerfile 模板（按语言挑一个）

> 这些模板**已经用了国内镜像源**，对应 [《镜像源速通指南》](https://hack-platform.jiarui.net.cn/docs/mirror) 里讲的优化。第一次写 Dockerfile 强烈建议直接抄。

### Node.js / Next.js

```
FROM node:20-alpine
WORKDIR /app


# 阿里 npm 源
RUN npm config set registry https://registry.npmmirror.com

COPY package.json package-lock.json* ./
RUN npm ci
COPY . .
RUN npm run build

EXPOSE 3000
CMD ["npm", "start", "--", "-H", "0.0.0.0", "-p", "3000"]
```

部署表单 port 填 **3000**。

### Python / FastAPI

```
FROM python:3.12-slim
WORKDIR /app
RUN sed -i 's|deb.debian.org|mirrors.aliyun.com|g' /etc/apt/sources.list.d/debian.sources \
 && pip config set global.index-url https://mirrors.aliyun.com/pypi/simple/

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .

EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

部署表单 port 填 **8000**。

### 纯静态 / SPA

```
FROM nginx:alpine
COPY dist/ /usr/share/nginx/html/
EXPOSE 80
```

部署表单 port 留空（用默认 80）。

## §4 环境变量与数据库凭证【重要，容易踩坑】

> ⚠️ 部署时**只有数据库凭证**由平台自动注入。**其他所有环境变量都必须写在你的 Dockerfile 里**（或在你的镜像内以别的方式带进去）。

### §4.1 数据库凭证（由平台自动注入）

每个队伍在活动开始前会收到一套独立的 PostgreSQL 数据库（内网访问 + 外网访问）。凭证会被平台**以 K8s Secret 的方式挂到你的容器**，你的应用代码里**直接读环境变量**就能连上。

可用的环境变量（部署到平台后自动就有）：

| 环境变量 | 内容 |
| --- | --- |
| `DATABASE_URL` | 完整的 `postgresql://user:pass@host:5432/dbname` DSN，**指向 VPC 内网 host（低延迟）** |
| `DB_HOST` | 只有 host（内网） |
| `DB_PORT` | 5432 |
| `DB_NAME` | 数据库名 |
| `DB_USER` | 用户名 |
| `DB_PASSWORD` | 密码 |

**应用里怎么用（举例）**：

```
// Node.js
const dsn = process.env.DATABASE_URL;
```

```


# Python
import os
dsn = os.environ["DATABASE_URL"]
```

**本地开发时**：把公网 DSN 写在项目根目录的 `.env`（**别提交到 git**），本地开发就用它连；部署到平台时，容器里的 `DATABASE_URL` 会被平台的 Secret 覆盖成 VPC 内网 DSN，**代码不用改**。

**⛔ 千万别做的事**：

-   ❌ 不要在 Dockerfile 里 `COPY .env`（会把公网 DSN 打进镜像，覆盖不掉平台注入的内网 DSN）

-   ❌ 不要在代码里硬编码 host / 用户名 / 密码
-   ❌ 不要把 `.env` 提交到 git（用 `.gitignore` 屏蔽）

**记得**：`.gitignore` 加一行 `.env`，Dockerfile 旁边的 `.dockerignore` 也加一行 `.env`。

---

### §4.2 其他环境变量（**必须**写在 Dockerfile 里）

平台**只注入数据库凭证**，不会注入你自己用的任何其他 key / URL / token。以下都算"其他"：

-   LLM API key（OpenAI、Anthropic、DeepSeek、智谱、Google、Groq …）

-   三方服务 token（Supabase、Sentry、Redis、MongoDB、S3/OSS ……）
-   应用配置常量（feature flag、时区、日志级别 ……）

-   前端 build-time 变量（`NEXT_PUBLIC_*`、`VITE_*` 等 —— 通常在 `npm run build` 时被硬编码到 bundle）

这些必须**在 Dockerfile 里直接写**（或让代码从代码里读默认值），例如：

```
FROM node:20-alpine
WORKDIR /app

# ⭐ 应用需要的所有其他环境变量都在这里写死
ENV OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxxxx
ENV DEEPSEEK_API_KEY=sk-yyyyyyyyyyyyyyyyyy
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1

# 数据库凭证不用写！平台会自动注入 DATABASE_URL / DB_HOST / DB_PORT / DB_NAME / DB_USER / DB_PASSWORD

COPY package.json package-lock.json* ./
RUN npm ci
COPY . .
RUN npm run build

EXPOSE 3000
CMD ["npm", "start", "--", "-H", "0.0.0.0", "-p", "3000"]
```

> 🔐 安全提示：镜像会推到平台的私有 ACR（只有你自己队伍能拉），所以 API key 写在 Dockerfile 里在这个活动里是**可接受**的。但**不要把带 key 的镜像 push 到 Docker Hub / GitHub 公开镜像仓库**。活动结束后建议轮换掉一次 key。

**为什么不允许别的方式注入**：K8s Deployment 目前只挂 `team-db` 这一个 Secret（envFrom），没有为参赛者应用预留额外的 Secret 挂载或 env 注入通道。48 小时活动为了简单起见就是这样的设计。

### §4.3 从公网 vs 内网连数据库

| 场景 | 用哪个地址 | 为什么 |
| --- | --- | --- |
| 你本机开发 | **公网 DSN**（`.env` 里放） | 你的电脑不在 VPC 里，只能走公网 |
| 部署到平台跑 | **内网 DSN**（平台自动注入） | 容器在 VPC 内网，走内网更快、更稳、免费 |

平台在你部署时**自动把 `DATABASE_URL` 换成内网 DSN**，你的应用无感切换。**这就是为什么代码里必须读 `process.env.DATABASE_URL`，不能硬编码。**

## §5 Build + Push

凭证和命令都在平台的「**部署**」页面，直接复制粘贴即可。这里只解释格式，不背记。

完整 push 命令长这样（XXX = 你的队伍编号）：

```


# 1. 登录 ACR（凭证在「部署」页面，每队不同）
docker login jzharbor-registry.cn-zhangjiakou.cr.aliyuncs.com \
  -u "LTAI..." -p "..."

# 2. 给本地镜像打 tag（注意是你的队伍编号 + 你想用的版本 tag）
docker tag my-local-image jzharbor-registry.cn-zhangjiakou.cr.aliyuncs.com/hack/team-XXX:v1

# 3. push
docker push jzharbor-registry.cn-zhangjiakou.cr.aliyuncs.com/hack/team-XXX:v1
```

> ⚠️ **你只能 push 到自己队伍的仓库**。tag 里的 `team-XXX` 必须跟你的队伍编号一致，不然 ACR 会拒绝（403）。

push 完，去「**部署**」页面的"第二步"，把 `v1`（或你打的任意 tag）填进去点部署。

## §6 常见翻车

| 现象 | 多半是 |
| --- | --- |
| **push 卡住 / 401** | 没登录 ACR；或登错了别的镜像仓库 |
| **push 403** | tag 里队伍编号写错了 |
| **部署后页面打不开 / 502** | 容器没监听 0.0.0.0，或者端口跟表单不一致 |
| **部署后一直 NotReady** | `GET /` 不返回 200。临时让 `/` 跳 home 或返回个 "ok" 就行 |
| **CrashLoopBackOff** | 容器启动直接挂掉。在平台「部署」页面查看 pod 日志 |
| **ImagePullBackOff** | 镜像 tag 写错，或 push 还没成功就触发了部署 |
| **OOMKilled** | 平台给每个 pod 的内存上限是 **2Gi**。Node.js 大项目 build 要在镜像内做完，运行时尽量瘦 |
| **连不上数据库 / `ECONNREFUSED` / `getaddrinfo ENOTFOUND`** | 见 §4。多半是 Dockerfile 里 `COPY .env` 把本地公网 DSN 打进了镜像，覆盖了平台注入的内网 DSN；或者代码硬编码了 host。修法：`.dockerignore` 加 `.env`，代码里只读 `process.env.DATABASE_URL` |
| **上线后 API 请求全 500 / 401** | 忘了在 Dockerfile 里 `ENV OPENAI_API_KEY=...` 之类，容器里没这个变量 |

## §7 调试技巧

-   **本地先跑一遍**：`docker run --rm -e DATABASE_URL="postgres://..." -e OPENAI_API_KEY=sk-... -p 你的端口:你的端口 镜像名` —— 浏览器打开 `http://localhost:你的端口` 看能不能正常出页面。本地不通的，平台上也不会通。用 `-e` 显式传 env，模拟平台注入 `DATABASE_URL` 的效果。

-   **多打几个 tag**：发现 bug 不要覆盖原来 tag，改成 `v2 v3 v4 …`。部署历史能让你看哪一版是好的，遇到回退也快。
-   **资源上限**：CPU 1 核 / 内存 2Gi。如果你的应用启动就吃满，请优化或拆分。

-   **验证平台环境变量是否真的注入了**：临时改一下你的 `/` 或加一个 `/env` 端点返回 `Object.keys(process.env).filter(k => k.startsWith("DB")).join(",")`，部署上去看看。看到 `DB_HOST,DB_NAME,DB_PASSWORD,DB_PORT,DB_USER` 就说明数据库 Secret 挂载正常。**上完线记得删掉这个端点**。
