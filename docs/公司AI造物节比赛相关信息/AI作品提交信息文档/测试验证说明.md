# 测试验证说明

## 1. 文档目标

本文用于指导自动化浏览器测试 agent 验证 `人力招聘数据分析后台` 的核心业务链路。测试重点是通过真实浏览器访问页面、执行筛选和跳转、读取页面可见数据，确认核心看板和目标达成场景可用。

测试不追求过细的 UI 像素检查，也不设置响应时间类性能指标。自动化测试应关注页面是否能正常访问、核心筛选是否可操作、关键数据是否正确展示、重要明细是否可以打开。

## 2. 测试范围和边界

### 2.1 访问入口

浏览器测试统一访问公网首页：

```text
https://hack-team-030.jiarui.net.cn/dashboard/overview
```

本站没有登录功能，所有页面公网可访问，不需要账号、密码、验证码或 SSO。

### 2.2 测试月份

数据库已存在 `2026-01` 至 `2026-05` 的全套数据。所有自动化浏览器测试默认选择并验证 `2026-05` 数据。

如果页面默认月份不是 `2026-05`，测试 agent 应通过页面上的 `月份` 控件选择 `2026-05` 后点击 `查询`。不同页面的月份控件可能表现为原生月份输入框或下拉搜索控件，测试时以页面实际控件为准。

### 2.3 只读验证要求

自动化测试只做只读验证，禁止污染真实数据。

允许执行：

- 打开页面和导航菜单。
- 修改筛选条件并点击 `查询`、`重置`。
- 点击看板跳转链接。
- 点击 `查看`、指标数字或表格行以打开明细弹窗。
- 读取页面文本、表格、卡片、弹窗内容。

禁止执行：

- 上传文件。
- 提交导入表单。
- 覆盖、清空、删除或修改真实业务数据。
- 使用导入类入口做试探性操作，例如 `招聘目标导入`、`数据导入`。
- 为了构造数据而调用任何写入型能力。

### 2.4 技术框架说明

本项目是服务端渲染应用，主要技术框架如下：

- 运行环境：`Node.js >= 24`。
- Web 框架：`Express`。
- 模板引擎：`EJS`。
- 页面样式：`Bootstrap` 加项目自定义 CSS。
- 数据库：`PostgreSQL`。

测试 agent 不应按前后端分离项目理解本系统。页面主要由服务端渲染输出，核心验证应通过浏览器访问页面并读取页面可见内容完成，不要假设存在稳定的纯后端 JSON API 可供直接断言。

## 5. 自动化浏览器测试用例

### 5.1 全站可访问和导航

操作步骤：

1. 打开公网首页 `/dashboard/overview`。
2. 确认页面标题或侧边栏包含 `人力招聘数据分析后台`。
3. 确认左侧导航包含 `综合分析看板`、`需求管理`、`面试记录管理`、`员工数据管理`。
4. 依次点击核心页面入口：`人才开发运营看板`、`基地风险分析`、`自主社招人效`、`目标达成进度`。

通过标准：

- 页面均可正常打开，不出现浏览器错误页或服务异常页。
- 当前页面的面包屑或标题与导航入口一致。
- 页面主体不是空白，能看到筛选区和业务内容区。

### 5.2 人才开发运营看板

操作步骤：

1. 打开 `/dashboard/overview`。
2. 选择月份 `2026-05` 并查询。
3. 读取 `运营风险摘要`、顶部指标卡、`整体目标达成`、`5 个渠道占比`、`渠道社招 TOP3`、`自招人效` 区域。
4. 点击 `进入风险诊断` 或重点风险基地链接，确认可以跳转到 `基地风险分析`。

通过标准：

- `运营风险摘要` 能展示包含目标、实际入培、达成率、GAP 的自然语言摘要。
- 顶部指标卡包含 `目标达成率`、`自主社招占比`、`自招7天人效`、`自招团队规模`。
- `2026-05` 参考指标应接近：目标达成率 `89.86%`、自主社招占比 `57.74%`、自招 7 天人效 `4.8`、团队规模 `28`。
- `整体目标达成` 中能看到多个基地，并能看到 `江苏基地-淮安` 的目标、实际、GAP 和达成率。
- 渠道占比合计口径应对应实际入培 `381`，并包含 `自主社招 220 · 57.74%`。

### 5.3 基地风险分析

操作步骤：

1. 打开 `/dashboard/base-risk`。
2. 选择月份 `2026-05` 并查询。
3. 不主动选择渠道，观察默认筛选和页面主体。
4. 如默认基地不是 `江苏基地-淮安`，可选择 `江苏基地-淮安` 后查询，验证该基地风险信息。
5. 点击不同批次卡片，观察右侧 `漏斗诊断与提升方案` 是否随批次切换。

通过标准：

- 页面包含 `月份`、`基地`、`渠道` 筛选项。
- 默认渠道应保持 `全部`，不应自动跳成 `自主社招`。
- 页面展示岗位月目标、已入职、GAP、达成率和风险批次。
- `江苏基地-淮安` 在 `2026-05` 下应接近：岗位月目标 `150`、已入职 `103`、GAP `-47`、达成率 `68.67%`。
- `漏斗诊断与提升方案` 区域不为空，能展示渠道状态、到面人数、面通人数、参培人数、诊断说明和提升方案。

### 5.4 自主社招人效

操作步骤：

1. 打开 `/dashboard/self-sourcing`。
2. 选择月份 `2026-05` 并查询。
3. 读取 `核心指标` 区域。
4. 查看 `自主社招招聘专员达成` 表格。
5. 点击表格中的参培达成、7 天人效或可点击明细数字，确认弹窗可以打开并展示员工明细。
6. 使用 `招聘专员` 筛选选择任意一名已有招聘专员并查询，再点击 `重置`。

通过标准：

- `核心指标` 包含整体人效、试用期人效、正式期人效、7 天人效达标人数、招聘团队规模。
- `2026-05` 参考指标应接近：整体 7 天人效 `4.8`、整体参培人效 `7.9`、招聘团队规模 `28`。
- 招聘专员表格不为空，至少展示姓名、员工状态、参培目标、参培达成、7 天人效、达成率等列。
- 明细弹窗可以打开和关闭，弹窗中能看到员工姓名、招聘渠道、渠道名称、入培日期、是否在职等字段。

## 7. 核心业务规则

### 7.1 一线员工对标目标

- 数据源优先使用 `common_emp_org_real_day`。
- 组织字段使用 `org_path`、`org_name`、`org_level2` 至 `org_level7`、`work_location` 共同推断基地。
- 基地归一后再对标目标表中的 `基地`。
- 目标对标维度为 `月份 + 基地 + 招聘渠道`。
- 人数按 `工号` 去重。
- 日期需先标准化为 `YYYY-MM-DD`，再判断月份。
- 组织路径包含 `江苏` 和 `淮安` 时，归一为 `江苏基地-淮安`。
- 组织路径包含 `韶关基地` 时，归一为 `南二在线客服项目`。
- 组织路径包含 `人才开发部` 时，用于识别招聘团队相关人员。

### 7.2 招聘专员规模和人效

- 招聘专员规模统计组织路径包含 `人才开发部` 的人员。
- 招聘专员岗位包括 `招聘专员` 和 `初级招聘主管`。
- 统计月内与在岗时间存在交集的人计入规模，统计月之前已离职人员不计入。
- 统计月最后一天入职满 6 个月为 `正式期`，未满 6 个月为 `试用期`。
- 自主社招候选人只统计 `招聘渠道 = 自主社招` 的一线员工。
- 候选人归属按 `渠道名称` 中 `+` 前的姓名匹配招聘专员姓名。
- `参培人效 = 参培达成 / 招聘规模`。
- `7天人效 = 满7天达成 / 招聘规模`。
- 满 7 天达成要求 `入培时间 + 7 天 <= 统计截止日`，且 7 天节点前未离职。
- 综合看板和自招人效页主口径使用 `7天人效`，`参培人效` 作为辅助指标展示。

## 8. 自动化测试执行建议

- 优先使用用户可见文本和语义化控件定位，例如页面标题、导航文字、表单标签、按钮文字、表格表头。
- 指标断言允许适度宽松，建议优先判断核心数字是否存在并接近参考值，不要依赖过细的 DOM 层级或 CSS 类名。
- 遇到公网版本尚未更新导致指标与本地参考值不一致时，应记录当前页面实际值，并标注为环境版本差异，不要误判为浏览器自动化操作失败。
- 如果页面出现空白、服务异常、核心筛选缺失、核心表格为空、弹窗无法打开，应判定为失败。
- 如果只是文案顺序、卡片排列或轻微样式变化，不应判定为失败。
- 测试过程中不要调用或构造非页面导航中的接口地址，避免把实现细节当成稳定测试契约。
